iOS闪退(崩溃)手动符号化解析

Philip.Lee

2016-04-11

iOS

解决闪退（崩溃）问题是移动应用开发者必备技能之一。
解决闪退可以分为线上和线下两种。线下（开发阶段）遇到闪退，可以直接DEBUG重现闪退。这个比较好解决。
但是线上APP的闪退，我们则只能收集闪退日志来分析。所有有可能会使用第三方闪退日志收集平台（如友盟等）或者直接使用AppStore的app分析也比较直接方便的。但是往往线上解析的crashLog并没有达到我的预期，没有完全解析出来（只有一个地址）。

这里介绍下自己手动再解析闪退日志的方式。解析闪退日志之前需要准备的东西:
(1) .dSYM(debugging SYMbols)又称为调试符号表
(2) .app（安装包）
(3) symbolicatecrash（解析闪退程序）

Apple自己的闪退日志的格式

这两段闪退日志是源于同一个iPA包，同一段代码。 我们可以看到Apple自身的没有解析出来的，会有两个地址 0x1000b4184 和0x100030000。而TalkingData上确是没有后一个地址的。 乍一看，他们的地址也是不一样的。
这个就需要说Apple引入的ASLR技术了

ASLR

ASLR 技术Address space layout randomization,ASLR通过将系统可执行程序随机装载到内存里，从而防止缓冲区溢出攻击
由于 ASLR 的缘故，导致程序crash后生成的crash log 中的 stack address 与 对应的 symbol address 不一致，有一个偏移量 slide，slide是程序装在时随机生成的随机数。

Stack address ： 程序运行时线程栈中所在函数调用的地址

Symbol address ： dsym文件中函数符号对应的地址，用此地址 在 dsym 文件中可以 查出对应的 符号信息。

Binary Image address(App Load Address): 程序装载的首地址

Slide Value： 在ASLR机制下每次启动APP装在之前 ，会在连接时指定的进程空间上这个偏移量就是 slide。

有ASLR机制时 Symbol address ＝ Stack Address + Slide Value - Binary Image Address

无ASLR机制时 Stack Address 等于Symble Address

这些地址如何去查找呢?

Stack Address：

就是程序闪退的地方，你可以在闪退日志中Last Exception Backtrace 找到。 以下的日志片段显示了程序闪退时方法调用的堆栈信息。日志片段中可以知道地址0x10014c290就是我们要找的Stack Address 堆栈地址

Binary Image address：

就是程序加载的其实地址，也有说是 Load Address，每次都有可能不一样。 在闪退日志找到 Binary Images: 部分，可以看到。该次APP启动占用的内存地址为 0x1000fc000-0x100563fff. 所以Binary Image address 是0x1000fc000
