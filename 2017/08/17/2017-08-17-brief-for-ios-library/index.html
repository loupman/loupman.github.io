静态库和动态库简析

Philip.Lee

2017-08-17

iOS

上一次我们讲了编译PB，把PB文件作为一个静态库，但是没有具体讲怎么抽取为一个静态库，加入其它项目时，需要注意些什么。这一篇主要介绍iOS中的静态库的使用和一些原理

静态库和动态库

存在形式

静态库： .a 和 .framework 的形式加入项目中
动态库： .dylib 和 .framework 的形式加入项目中

使用上的区别

静态库：链接时，静态库会被完整地复制到可执行文件中， 被多次使用就有多份冗余拷贝
动态库：链接时不复制，程序运行时由系统动态加载到内存，供程序调用，（系统动态库）系统只加载一次，多个程序共用，节省内存。
在iOS8以前一般系统库才会是动态库，第三方库一般是静态库，如果项目中使用了自制的动态库，不能被上传到 AppStore会被拒的。iOS8以后，Apple开放了用户也可以使用动态库了，但是是私用的，不能共享。

动态库

静态库在编译时就和APP代码链接并打包进同一个二进制包中了，动态库可以在运行时手动加载。我们可以在以下几个场景中使用了。

    1. 应用插件化

商户端的APP首页完全可以做成插件化管理。 超市，订单，工作，业绩中心，完全可以做成一个个插件来管理。

    1. 软件版本实时模块升级

前提是这个功能模块是使用动态库加载实现的。如果软件中的某个功能模块出现了严重的BUG，或者想在其中新增功能，这时候你只需要在适当的时候从服务器上将新版本的动态库文件下载到本地，然后在用户重启应用的时候即可实现新功能的展现。

    1. 共享可执行文件，节省内存

iOS仍然不允许进程间共享动态库，即iOS上的动态库只能是私有的，因为我们仍然只能将动态库文件存入自身沙盒里，其他应用程序无法访问。

怎么区分 .framework 是动态库还是静态库呢？
一般新建 Cocoa Touch Framework 项目就是动态库了。 Cocoa Touch Static Library 项目就是静态库了。在项目中也是可以查看库是静态库的还是动态库

新建方式： File -> New -> Project (⇧⌘ + N)

这种方式一般是引入第三方库的时候，一般包括 .h 和 .a 或者 .framework

打包静态库

打包静态库，有四种版本

  • 真机-Debug版本

  • 真机-Release版本

  • 模拟器-Debug版本

  • 模拟器-Release版本

如下图
四种的大小也是个不一样。其中真机-Release版本的最大。模拟器-Debug版本的最小。 这和预想的好像不一致。我们使用以下命令来看看生成的 .a 所支持的CPU指令集。

当选择Build Active Architecture Only 是默认选择时（Debug为YES， Release 为NO）

真机-Debug 真机-Release 模拟器-Debug 模拟器-Release
arm64 armv7 arm64 i386 x86_64 i386 x86_64
6MB 10.5MB 5.3MB 9.9MB

** Architecture ： 指你想支持的指令集。
Valid architectures : 指即将编译的指令集。
Build Active Architecture Only : 是否只编译当前适用的指令集。
最后编译的包支持的指令集是 Architecture 和 Valid architectures 两者的交集 **

我们发现他们所导出的包的指令集不一样，这样导致包的大小也不一致的主要原因。 因为指令集都是向下兼容的， arm64 > armv7s > armv7
要发挥支持64位指令集的手机（如iPhone6）的64位性能，就要包含64位包，那么系统最低要求为ios6。 只能打32位的包，系统都能通用，但是会丧失支持64位指令集的手机（如iPhone6）的性能。不能打只支持64位指令集的静态库或者动态库的包，否者只支持32位指令集的手机就不支持这个静态库和动态库了。

下面是不同设备对应的架构：
armv7：iPhone 3GS/4/4s，iPod 3G/4G，iPad 1G/2G/3G （32位）
armv7s：iPod5, iPhone5, iPhone5C （32位）
arm64: iPhone5s, iPhone6, iPhone6s （64位）

当把 Build Active Architecture Only都设置为 NO时，这时Release模式的包确实比Debug模式的包小一点。 即使用同一种CPU指令架构来编译Release的包会比Debug模式的包小。
